# Code of Conduct

> **Version 1.0** | **Last Updated:** 2026-04-03 | **Contact:** hello.finksystems@gmail.com

---

## Our Commitment

MCP Verify is a security-focused open-source project built on principles of **transparency**, **responsibility**, and **collaboration**. We are committed to fostering a welcoming, inclusive, and harassment-free environment for everyone—regardless of age, body size, disability, ethnicity, gender identity, level of experience, nationality, personal appearance, race, religion, or sexual orientation.

Security work requires trust, and trust is built through respectful, professional interactions. We expect all community members to uphold these standards.

---

## Our Values

As a security project, we hold ourselves to higher standards:

### 🔒 Security Responsibility

- **Responsible disclosure**: Security vulnerabilities must be reported privately (see [Reporting Process](#reporting-process))
- **No weaponization**: Do not use mcp-verify for malicious purposes, unauthorized testing, or attacks
- **Ethical research**: Only test systems you own or have explicit written authorization to test
- **No exploitation**: Do not share exploits publicly before patches are available

### 🤝 Community Collaboration

- **Constructive feedback**: Critique ideas, not people
- **Knowledge sharing**: Help newcomers learn security best practices
- **Credit where due**: Acknowledge contributions and prior work
- **Open communication**: Assume good intentions, seek clarity before conflict

### 📚 Transparency & Integrity

- **Honest reporting**: Do not exaggerate vulnerabilities or create false positives
- **Clear documentation**: Make security findings understandable to both technical and non-technical audiences
- **Admit mistakes**: We all make errors—own them, fix them, learn from them

---

## Expected Behavior

We expect all community members to:

### ✅ In All Interactions

- Use welcoming, inclusive language
- Respect differing viewpoints and experiences
- Accept constructive criticism gracefully
- Focus on what's best for the community and project
- Show empathy toward other community members

### ✅ In Technical Discussions

- Provide evidence-based security analysis
- Distinguish between theoretical risks and practical exploits
- Use severity levels accurately (don't label everything "critical")
- Share reproducible steps for reported issues
- Respect embargo periods for security vulnerabilities

### ✅ In Code Contributions

- Follow the project's coding standards (see [DEVELOPMENT.md](./DEVELOPMENT.md))
- Write clear, self-documenting code with appropriate comments
- Include tests for new features and bug fixes
- Sign-off commits with Developer Certificate of Origin (DCO)
- Respect the AGPL-3.0 license terms

## Unacceptable Behavior

The following behaviors are **not tolerated** in this community:

### ❌ Harassment & Discrimination

- Sexualized language or imagery, unwelcome sexual attention
- Trolling, insulting/derogatory comments, personal or political attacks
- Public or private harassment, intimidation, or stalking
- Publishing others' private information (doxxing) without consent
- Deliberate misgendering or use of rejected names

### ❌ Security Misconduct

- **Public disclosure of vulnerabilities** without maintainer coordination
- **Weaponizing findings** for extortion, blackmail, or unauthorized access
- **False positives spam** (deliberately submitting junk reports)
- **Supply chain attacks** (malicious contributions, backdoors)
- **Social engineering** community members for credentials or access

### ❌ Professional Misconduct

- Plagiarism or claiming others' work as your own
- Sock-puppeting (creating fake accounts to support your arguments)
- Spamming, advertising unrelated products/services
- Disrupting discussions with off-topic or inflammatory content
- Demanding unpaid labor from maintainers ("fix this now!")

### ❌ Other Conduct

- Any conduct that could reasonably be considered inappropriate in a professional setting
- Advocating for or encouraging any of the above behaviors

## Reporting Process

### 🔒 Security Vulnerabilities (Private)

**DO NOT** open public issues for security vulnerabilities in mcp-verify or servers tested with mcp-verify.

**Report to:** hello.finksystems@gmail.com

**Include:**

- Description of the vulnerability
- Steps to reproduce (with minimal test case if possible)
- Impact assessment (what can an attacker achieve?)
- Suggested remediation (optional, but appreciated)

**We commit to:**

- Acknowledge receipt within **48 hours**
- Provide status updates every **7 days** until resolution
- Credit reporters in security advisories (unless anonymity requested)
- Coordinate public disclosure timing with reporters

**Responsible disclosure timeline:**

- **90 days** from initial report to public disclosure (industry standard)
- Earlier disclosure if actively exploited in the wild
- Extended timeline if fix requires significant architectural changes

---

### 📢 Code of Conduct Violations (Confidential)

**Report to:** hello.finksystems@gmail.com

**Include:**

- Your contact information (for follow-up questions)
- Names of individuals involved
- Description of what happened (with links/screenshots if available)
- When and where it occurred
- Any other relevant context

**Confidentiality:**

- All reports are treated confidentially
- We will not disclose reporter identity without consent
- Reports are shared only with maintainers on a need-to-know basis

**Response timeline:**

- **Acknowledgment:** Within 48 hours
- **Initial review:** Within 7 days
- **Resolution decision:** Within 30 days (complex cases may take longer)

## Enforcement

### Investigation Process

1. **Report received** → Maintainers review and acknowledge receipt
2. **Fact-finding** → Collect evidence, interview involved parties
3. **Deliberation** → Maintainers discuss appropriate response
4. **Decision** → Communicate outcome to reporter and involved parties
5. **Action** → Implement consequences (if applicable)
6. **Follow-up** → Monitor to ensure behavior changes

### Consequences

Depending on severity and context, consequences may include:

| Severity     | Violation Examples                                                 | Consequences                                          |
| ------------ | ------------------------------------------------------------------ | ----------------------------------------------------- |
| **Minor**    | Unintentional insensitive language, first-time minor rudeness      | Private warning with explanation                      |
| **Moderate** | Repeated minor violations, public arguments, derailing discussions | Temporary ban (7-30 days) from community spaces       |
| **Serious**  | Harassment, doxxing, public vuln disclosure without coordination   | Permanent ban, removal from contributor list          |
| **Critical** | Supply chain attacks, weaponization, extortion                     | Permanent ban, legal action, law enforcement referral |

**Note:** Maintainers have final discretion on enforcement. We aim for **rehabilitation over punishment** when appropriate, but community safety is the priority.

## Appeals Process

If you believe a moderation decision was made in error:

1. **Email:** hello.finksystems@gmail.com with subject "Appeal: [Brief Description]"
2. **Include:**
   - Original incident details
   - Why you believe the decision was incorrect
   - Any new evidence not previously considered
3. **Timeline:** Appeals reviewed within **14 days**
4. **Decision:** Appeal decisions are **final**

**Note:** Abusing the appeals process (frivolous appeals, harassment during appeal) may result in additional consequences.
